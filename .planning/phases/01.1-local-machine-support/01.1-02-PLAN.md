**`api/machines.py`**: Modify the machine list and related endpoints to include "This Machine":

1. Add imports: `from app.local.manager import LOCAL_MACHINE_ID, LOCAL_MACHINE_NAME, local_machine_manager` and `from app.services.machine_registry import is_local_machine, get_connection_for_machine`.

2. Create a helper function `_local_machine_response()` that returns a `MachineResponse` for the local machine:
   - `id=LOCAL_MACHINE_ID` (the string "local")
   - `name=LOCAL_MACHINE_NAME` ("This Machine")
   - `host="localhost"`, `port=0`, `username=""`, `ssh_key_path=""`
   - `repo_scan_paths=settings.local_repo_scan_paths` (from config)
   - `status="online"` (always)

3. Modify `list_machines()`: Prepend the local machine response before the DB machines list. Return `[_local_machine_response()] + [_machine_to_response(m) for m in machines]`.

4. Modify `get_machine()`: Add early check -- if `str(machine_id) == LOCAL_MACHINE_ID`, return `_local_machine_response()` instead of DB lookup.

5. Modify `scan_repos()`: Add early check for local machine. If `str(machine_id) == LOCAL_MACHINE_ID`, use `local_machine_manager.run_command()` to run the find command, using `settings.local_repo_scan_paths`. Return results. Skip the DB lookup and SSH connection path.

6. Modify `get_tmux_sessions()`: Add early check for local machine. If local, get connection from local_machine_manager. If connection exists (Docker mode), use existing `list_tmux_sessions(conn)`. If no connection (native mode), use `list_tmux_sessions_local()` from `app.local.tmux`. Return same response format.

7. Modify `create_tmux_session()`: Add early check for local machine. Similar routing as get_tmux_sessions -- use SSH-based or subprocess-based tmux creation based on Docker mode.

8. Do NOT add local machine to create, update, delete, connect, disconnect, or test-connection endpoints. "This Machine" is synthetic and cannot be CRUD'd or connection-managed. If someone tries to DELETE or PUT machine_id="local", return 400 with "Cannot modify the local machine" message.

**`api/sessions.py`**: Modify session creation to handle local machine:

1. Modify `create_session()`: If `body.machine_id == LOCAL_MACHINE_ID`, skip the machine DB lookup (there's no Machine row for local). Create the TerminalSession directly with `machine_id=LOCAL_MACHINE_ID`. The `machine_id` is now a string field (Plan 01 changed this).

2. Modify `delete_session()`: When killing a session on the local machine, route tmux kill through the registry. If `machine_id == LOCAL_MACHINE_ID`:
   - Get connection via `local_machine_manager.get_connection()`
   - If connection exists (Docker mode), use existing SSH-based `kill_tmux_session(conn, tmux_name)`
   - If no connection (native mode), use `kill_tmux_session_local(tmux_name)` from `app.local.tmux`
   - The death animation: for Docker mode, use SSH to run it as before. For native mode, skip the death animation (subprocess tmux pane injection is complex and not essential). OR run it via subprocess writing to the pane TTY -- use your judgment, but don't let this block the plan. A simple approach is fine.

3. In the delete endpoint, update the `conn = await ssh_manager.get_connection(machine_id)` call to use `get_connection_for_machine(machine_id)` from the registry instead. Same for the `detach_session` endpoint.

4. In `list_sessions()` -- no change needed. Sessions for the local machine will have `machine_id="local"` and will be returned by the existing query.

**`main.py`**: Add local machine initialization to the lifespan:

1. Import `local_machine_manager` from `app.local.manager`.

2. After the DB table creation and before the SSH auto-connect loop, call `await local_machine_manager.initialize()`. This will SSH to the host in Docker mode or no-op in native mode.

3. In the shutdown section (after yield), call `await local_machine_manager.shutdown()` before `ssh_manager.shutdown()`.

4. Log the local machine initialization result: "Local machine: initialized (Docker mode: SSH to host)" or "Local machine: initialized (native mode: subprocess)".

**`ws/terminal.py`**: Modify the terminal WebSocket handler to support local machine sessions:

1. Add imports: `from app.local.manager import LOCAL_MACHINE_ID, local_machine_manager` and `from app.services.machine_registry import is_local_machine`.

2. Modify `_get_or_create_process()`:
   - Change the function signature: remove the `conn` parameter. Instead, take `machine_id` as a parameter and resolve the connection inside the function.
   - After the existing pool check (returning alive existing process), add routing logic:
     - If `is_local_machine(machine_id)`:
       - Get connection via `local_machine_manager.get_connection()`
       - If connection exists (Docker mode): use existing `create_terminal_in_tmux(conn, ...)` -- same as remote
       - If no connection (native mode): import and use `create_local_terminal_in_tmux()` from `app.local.tmux`. This returns a `LocalTerminalProcess` which has the same interface as AsyncSSH process.
     - Else (remote machine): use existing `ssh_manager.get_connection(machine_id)` path -- unchanged logic
   - The `SessionProcess` class should work unchanged with `LocalTerminalProcess` because it uses the same `.stdout.read()`, `.stdin.write()`, `.channel.set_terminal_size()`, `.close()`, `.wait_closed()` interface.

3. Modify `terminal_websocket()`:
   - For local machine sessions, skip the `ssh_manager.get_connection()` check. Instead:
     - If `machine_id == LOCAL_MACHINE_ID`: the machine is always "connected", proceed directly
     - If remote: keep existing SSH connection check
   - Update the `_get_or_create_process()` call to pass `machine_id` instead of `conn`.

4. The `SessionProcess` class itself should NOT need changes. It already uses duck-typed process interface (`.stdout.read()`, `.stdin.write()`, `.channel.set_terminal_size()`). The `LocalTerminalProcess` from Plan 01 implements this same interface.

5. The `close_session_process()` function needs no changes -- it just removes from the pool and calls `.close()`.

**`ws/status.py`**: Include local machine in status polling and Claude detection:

1. Add imports: `from app.local.manager import LOCAL_MACHINE_ID, LOCAL_MACHINE_NAME, local_machine_manager`.

2. Modify `_build_initial_snapshot()`:
   - Add local machine to `machine_statuses`: `machine_statuses[LOCAL_MACHINE_ID] = "online"`
   - Detect Claude sessions on local machine:
     - Get connection via `local_machine_manager.get_connection()`
     - If connection exists (Docker mode): reuse `_detect_session_statuses(LOCAL_MACHINE_ID, conn)` -- same as remote
     - If no connection (native mode): create a `_detect_local_claude_sessions()` helper that uses subprocess to run `tmux list-windows -a -F '...'` and parses for claude processes. Reuse the parsing logic from `detect_claude_sessions()` but with subprocess instead of SSH.

3. Modify `_poll_loop()`:
   - In the machine status polling section (every Nth tick), also send local machine status (always "online")
   - In the Claude sessions polling section, add local machine Claude detection alongside the SSH machine loop. Append local Claude sessions to `all_claude`.

4. Modify `_poll_session_names()`:
   - For sessions with `machine_id == LOCAL_MACHINE_ID`, get the connection or use subprocess to detect pane display name.
   - If Docker mode: reuse existing `_detect_pane_display_name(conn, tmux_name)`
   - If native mode: create a subprocess-based equivalent that runs `tmux display-message -p -t '{name}' '#{pane_current_path}'` and `git rev-parse --abbrev-ref HEAD`

**`services/claude.py`**: Add a subprocess-based alternative for native mode Claude detection:

1. Add a new function `async detect_claude_sessions_local() -> list[dict]`:
   - Uses `asyncio.create_subprocess_exec("tmux", "list-windows", "-a", "-F", "#{session_name}:#{window_index}:#{window_name}:#{pane_current_command}")`
   - Parses output identically to the SSH version
   - Catches `FileNotFoundError` (no tmux installed) and returns empty list
   - Returns same dict format: `tmux_session`, `window_index`, `window_name`, `command`, `status`

2. Add `async detect_claude_session_status_local(tmux_session: str, window_index: int) -> str`:
   - Subprocess version of `detect_claude_session_status()`
   - Checks marker file via `cat /tmp/.locus-claude-status`, same parsing logic
   - Falls back to subprocess-based `tmux capture-pane -t {session}:{window} -p | tail -5` for waiting detection
   - Returns "idle", "running", or "waiting"
