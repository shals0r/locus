## Decisions Made
- **is_usable property**: Single boolean check on LocalMachineManager rather than scattering `in_docker and conn is None` checks everywhere. Clean guard that every code path can use.
- **409 Conflict for needs_setup**: Matches existing pattern where remote machines return 409 when "not connected". Semantically correct -- the machine exists but can't serve requests.
- **Hide SessionTabBar entirely**: Instead of disabling the "+" button, the CenterPanel doesn't render SessionTabBar at all when needs_setup. Cleaner UX -- no confusing disabled buttons.
- **ConnectionError from run_command**: In Docker mode without SSH, run_command raises instead of silently executing in the container. Prevents any accidental container-side execution.

## Deviations from Plan

The original plan had 2 tasks: (1) frontend local machine support, (2) human verification checkpoint. During checkpoint verification, the user discovered the fundamental Docker container shell problem. The continuation implements "honest degradation" as directed by the user's checkpoint feedback -- this is not an auto-fix deviation but rather the checkpoint system working as designed.

No Rule 1-4 auto-fix deviations occurred.

## Issues Encountered
- **Docker container shell problem**: The original design assumed `conn=None` means "use subprocess" universally. In Docker mode, subprocess runs inside the container, not on the host. The fix adds the `is_usable` guard that distinguishes "native mode subprocess" (correct) from "Docker mode without SSH" (incorrect). Discovered during checkpoint verification.
