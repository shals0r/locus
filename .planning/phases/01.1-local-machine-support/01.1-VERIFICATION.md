# Phase 1.1 Verification: Local Machine Support

## Status: PASSED

## Must-Have Verification

### 1. "This Machine" appears automatically — no manual setup, always present
**PASS** — `LOCAL_MACHINE_ID = "local"` defined in `backend/app/local/manager.py:14`. `list_machines()` in `backend/app/api/machines.py` prepends local machine response before DB machines. Frontend `machineStore.ts` sorts local machine first. `MachineTab.tsx` shows Monitor icon for local machine.

### 2. Terminal tabs on local machine use local PTY/tmux (not SSH-to-localhost)
**PASS** — `backend/app/local/terminal.py` implements `LocalTerminalProcess` with `pty.openpty()` + `os.fork()`. `backend/app/local/tmux.py` provides `create_local_terminal_in_tmux()` via subprocess. `backend/app/ws/terminal.py` routes via `is_local_machine()` check, uses subprocess path when `conn is None` (native mode).

### 3. Local tmux sessions visible and attachable
**PASS** — `list_tmux_sessions_local()` and `check_tmux_session_exists_local()` in `backend/app/local/tmux.py`. `backend/app/api/machines.py` routes tmux endpoints through local path. Terminal WebSocket reattach path handles native mode with `check_tmux_session_exists_local()`.

### 4. Local Claude Code sessions detected
**PASS** — `detect_claude_sessions_local()` in `backend/app/services/claude.py:135` uses subprocess to parse tmux windows. `backend/app/ws/status.py` includes local machine in Claude detection polling.

### 5. Local repo discovery via filesystem scan
**PASS** — `scan_repos()` in `backend/app/api/machines.py` handles `LOCAL_MACHINE_ID` with `local_machine_manager.run_command()`. Config has `local_repo_scan_paths` setting.

## Honest Degradation (Docker Mode Without SSH)

When running in Docker without host SSH configured:
- `LocalMachineManager.get_status()` returns `"needs_setup"` (not `"online"`)
- `is_usable` property returns `False`
- Terminal creation blocked with 409 in API and WebSocket handlers
- Frontend shows amber `AlertTriangle` icon and setup instructions
- Phase 5 (Host Agent) added to roadmap to properly solve Docker-to-host communication

This is intentional — prevents the incorrect behavior of giving a container shell when user expects host access.

## Notes

- All criteria fully met in **native mode** (backend on host) and **Docker+SSH mode**
- Docker-without-SSH mode has intentional degradation (not a gap — Phase 5 addresses this)
- `LocalTerminalProcess` duck-types AsyncSSH interface (`channel.set_terminal_size()`, `stdout.read()`, `stdin.write()`)
- Unix-only PTY gated behind `sys.platform != "win32"` check
