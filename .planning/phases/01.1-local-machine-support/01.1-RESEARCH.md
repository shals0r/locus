# Phase 1.1: Local Machine Support - Research

**Researched:** 2026-03-26
**Domain:** Local terminal access, PTY management, Docker-to-host connectivity
**Confidence:** HIGH

## Summary

Phase 1.1 adds "This Machine" as a first-class, always-connected machine in Locus. The primary challenge is that Locus runs in Docker, but "local machine" means the HOST machine, not the container. This creates a fundamental architectural tension: local PTY/subprocess calls inside the container only access the container's filesystem and processes, not the host's.

After investigating all viable approaches (local PTY via subprocess, nsenter with host PID namespace, Docker socket, SSH-to-host), the recommended architecture uses **SSH to `host.docker.internal`** as the transport layer when running in Docker, while providing a **native PTY path** when the backend runs directly on the host (for development or non-Docker deployments). The key design principle is a `LocalMachineManager` abstraction that provides the same interface as `SSHManager` but auto-detects whether it's running in Docker or natively, choosing the appropriate transport.

This approach maximizes code reuse (the existing SSH/tmux infrastructure handles 90% of the work), requires minimal frontend changes (the local machine just looks like another machine with special status), and gracefully handles the Docker constraint without requiring privileged containers or host PID namespace access.

**Primary recommendation:** Create a `LocalMachineManager` that uses SSH to `host.docker.internal` in Docker or `asyncio.create_subprocess_exec` natively. Surface "This Machine" as a synthetic machine entry that's always present, always "online", and never stored in the database.

## Standard Stack

### Core (No New Dependencies)

| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|
| AsyncSSH | ~2.22.0 (existing) | SSH to host from Docker container | Already used for remote machines. Same library handles local-via-SSH path. |
| asyncio subprocess | stdlib | Native PTY subprocess (non-Docker path) | Python stdlib. Used when backend runs directly on host. |
| pty | stdlib | Pseudo-terminal allocation (non-Docker path) | Python stdlib. Creates PTY pairs for terminal sessions. |

### Supporting (No New Dependencies)

| Library | Version | Purpose | When to Use |
|---------|---------|---------|-------------|
| os / fcntl | stdlib | PTY file descriptor management | Non-blocking I/O on PTY master fd |
| struct / termios | stdlib | Terminal size setting | TIOCSWINSZ ioctl for resize |

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| SSH to host.docker.internal | nsenter + host PID namespace | Requires --privileged and --pid=host. Major security concern. Breaks container isolation. |
| SSH to host.docker.internal | Docker socket + exec on host | Would need a sidecar process on host. Adds operational complexity. Docker socket access is a security risk. |
| SSH to host.docker.internal | ttyd/gotty sidecar on host | Requires installing and running a separate service on the host. Defeats "zero setup" goal. |
| asyncio subprocess (native) | libtmux Python library | Adds dependency. Raw tmux CLI via subprocess is simpler and matches existing SSH approach. |

**No new packages to install.** This phase uses only existing dependencies.

### Pattern 1: Transport Abstraction via LocalMachineManager

**What:** A class that mirrors `SSHManager`'s interface but handles the local machine. It auto-detects whether it's running in Docker (via `/.dockerenv` file or `LOCUS_IN_DOCKER` env var) and uses the appropriate transport.

**When to use:** Every time the system needs to execute a command on the local machine, list tmux sessions, create terminal processes, or scan for repos.

**Example:**

```python
# backend/app/local/manager.py

import asyncio
import os
import logging

from app.config import settings

logger = logging.getLogger(__name__)

# Well-known ID for the local machine (never stored in DB)
LOCAL_MACHINE_ID = "local"
LOCAL_MACHINE_NAME = "This Machine"


def is_running_in_docker() -> bool:
    """Detect if we're running inside a Docker container."""
    return os.path.exists("/.dockerenv") or os.environ.get("LOCUS_IN_DOCKER") == "1"


class LocalMachineManager:
    """Manages the local machine connection.

    When running in Docker: uses SSH to host.docker.internal
    When running natively: uses asyncio subprocess directly
    """

    def __init__(self) -> None:
        self._in_docker = is_running_in_docker()
        self._ssh_conn = None  # AsyncSSH connection (Docker mode only)
        self._status = "online"  # Always online

    @property
    def in_docker(self) -> bool:
        return self._in_docker

    async def initialize(self) -> None:
        """Set up the local machine connection on startup."""
        if self._in_docker:
            await self._connect_to_host()
        # Native mode: nothing to initialize

    async def get_connection(self):
        """Get connection object.

        Returns AsyncSSH connection in Docker mode,
        or None in native mode (callers use subprocess).
        """
        if self._in_docker:
            return self._ssh_conn
        return None  # Sentinel: use subprocess path

    async def run_command(self, command: str) -> str:
        """Run a command on the local machine and return stdout."""
        if self._in_docker and self._ssh_conn:
            result = await self._ssh_conn.run(command, check=True)
            return result.stdout
        else:
            proc = await asyncio.create_subprocess_shell(
                command,
                stdout=asyncio.subprocess.PIPE,
                stderr=asyncio.subprocess.PIPE,
            )
            stdout, _ = await proc.communicate()
            return stdout.decode()

    def get_status(self) -> str:
        return "online"  # Always online

    async def _connect_to_host(self) -> None:
        """SSH to the Docker host via host.docker.internal."""
        import asyncssh
        host = os.environ.get("LOCUS_HOST_SSH_HOST", "host.docker.internal")
        port = int(os.environ.get("LOCUS_HOST_SSH_PORT", "22"))
        username = os.environ.get("LOCUS_HOST_SSH_USER", "")
        key_path = os.environ.get("LOCUS_HOST_SSH_KEY", "")

        if not username or not key_path:
            logger.warning(
                "Local machine SSH not configured. Set LOCUS_HOST_SSH_USER "
                "and LOCUS_HOST_SSH_KEY for Docker-to-host access."
            )
            return

        try:
            self._ssh_conn = await asyncssh.connect(
                host, port=port, username=username,
                client_keys=[key_path], known_hosts=None,
                keepalive_interval=15, keepalive_count_max=3,
            )
            logger.info("Local machine: connected to host via SSH (%s@%s:%d)", username, host, port)
        except Exception as exc:
            logger.warning("Local machine: SSH to host failed: %s", exc)
            self._ssh_conn = None

    async def shutdown(self) -> None:
        if self._ssh_conn:
            self._ssh_conn.close()
            self._ssh_conn = None


local_machine_manager = LocalMachineManager()
```

### Pattern 2: Machine Registry (Unified Access)

**What:** A registry that provides a single lookup for any machine (local or remote), so API routes and WebSocket handlers don't need to know which type they're dealing with.

**When to use:** In API routes, WebSocket handlers, and status polling.

**Example:**

```python
# backend/app/services/machine_registry.py

from app.local.manager import LOCAL_MACHINE_ID, local_machine_manager
from app.ssh.manager import ssh_manager


async def get_connection_for_machine(machine_id: str):
    """Get connection for any machine (local or remote)."""
    if machine_id == LOCAL_MACHINE_ID:
        return await local_machine_manager.get_connection()
    return await ssh_manager.get_connection(machine_id)


def get_machine_status(machine_id: str) -> str:
    """Get status for any machine."""
    if machine_id == LOCAL_MACHINE_ID:
        return local_machine_manager.get_status()
    return ssh_manager.get_status(machine_id)


def is_local_machine(machine_id: str) -> bool:
    return machine_id == LOCAL_MACHINE_ID
```

### Pattern 3: Local Tmux via Subprocess (Native Mode)

**What:** When running natively (not in Docker), tmux operations use `asyncio.create_subprocess_exec` instead of SSH.

**Example:**

```python
# backend/app/local/tmux.py

import asyncio
import re
from uuid import uuid4


async def list_tmux_sessions_local() -> list[dict]:
    """List tmux sessions on the local machine."""
    try:
        proc = await asyncio.create_subprocess_exec(
            "tmux", "ls", "-F",
            "#{session_name}:#{session_attached}:#{session_activity}",
            stdout=asyncio.subprocess.PIPE,
            stderr=asyncio.subprocess.PIPE,
        )
        stdout, _ = await proc.communicate()
        if proc.returncode != 0:
            return []

        sessions = []
        for line in stdout.decode().strip().split("\n"):
            if not line:
                continue
            parts = line.split(":")
            if len(parts) >= 3:
                sessions.append({
                    "name": parts[0],
                    "attached": int(parts[1]) > 0,
                    "last_activity": int(parts[2]),
                })
        return sessions
    except FileNotFoundError:
        return []  # tmux not installed
```

### Pattern 4: Local PTY Terminal Process (Native Mode)

**What:** When running natively, terminal sessions use Python's `pty` module to create PTY pairs and spawn tmux processes directly.

**Example:**

```python
# backend/app/local/terminal.py

import asyncio
import fcntl
import os
import pty
import struct
import termios


class LocalTerminalProcess:
    """A local terminal process backed by a PTY.

    Provides the same read/write/resize interface as AsyncSSH processes
    so SessionProcess can use it interchangeably.
    """

    def __init__(self, master_fd: int, pid: int, cols: int, rows: int):
        self._master_fd = master_fd
        self._pid = pid
        self._closed = False
        # Make master fd non-blocking for asyncio
        flags = fcntl.fcntl(master_fd, fcntl.F_GETFL)
        fcntl.fcntl(master_fd, fcntl.F_SETFL, flags | os.O_NONBLOCK)
        self._set_size(cols, rows)
        # Create asyncio reader
        self._loop = asyncio.get_event_loop()
        self._reader = asyncio.StreamReader()

    def _set_size(self, cols: int, rows: int) -> None:
        """Set terminal size via ioctl."""
        winsize = struct.pack("HHHH", rows, cols, 0, 0)
        fcntl.ioctl(self._master_fd, termios.TIOCSWINSZ, winsize)

    # Stdout-like interface for SessionProcess compatibility
    @property
    def stdout(self):
        return self  # self acts as stdout reader

    @property
    def stdin(self):
        return self  # self acts as stdin writer

    async def read(self, n: int = 16384) -> bytes:
        """Read from PTY master fd."""
        loop = asyncio.get_event_loop()
        future = loop.create_future()

        def _readable():
            try:
                data = os.read(self._master_fd, n)
                if not future.done():
                    future.set_result(data)
            except OSError:
                if not future.done():
                    future.set_result(b"")
            loop.remove_reader(self._master_fd)

        loop.add_reader(self._master_fd, _readable)
        return await future

    def write(self, data: bytes) -> None:
        """Write to PTY master fd (stdin)."""
        if not self._closed:
            os.write(self._master_fd, data)

    @property
    def channel(self):
        return self  # self acts as channel for resize

    def set_terminal_size(self, cols: int, rows: int) -> None:
        self._set_size(cols, rows)

    def close(self) -> None:
        self._closed = True
        try:
            os.close(self._master_fd)
        except OSError:
            pass

    async def wait_closed(self) -> None:
        try:
            _, _ = os.waitpid(self._pid, os.WNOHANG)
        except ChildProcessError:
            pass


async def create_local_terminal_in_tmux(
    session_name: str | None = None,
    working_dir: str | None = None,
    cols: int = 120,
    rows: int = 40,
) -> tuple:
    """Create a local tmux terminal session with PTY.

    Returns (LocalTerminalProcess, tmux_session_name).
    """
    if session_name:
        command = f"tmux attach -t '{session_name}'"
    else:
        name = f"locus-{os.urandom(4).hex()}"
        cd_flag = f" -c '{working_dir}'" if working_dir else ""
        command = f"tmux new-session -s '{name}'{cd_flag}"
        session_name = name

    # Fork with PTY
    pid, master_fd = pty.openpty()
    child_pid = os.fork()

    if child_pid == 0:
        # Child process
        os.setsid()
        os.dup2(pid, 0)
        os.dup2(pid, 1)
        os.dup2(pid, 2)
        os.close(master_fd)
        os.close(pid)
        os.environ["TERM"] = "xterm-256color"
        os.execvp("bash", ["bash", "-c", command])
    else:
        os.close(pid)
        process = LocalTerminalProcess(master_fd, child_pid, cols, rows)
        return process, session_name
```

### Pattern 5: Synthetic "This Machine" Entry

**What:** "This Machine" is NOT stored in the database. It's injected at the API level into machine lists and handled specially by the frontend.

**When to use:** Machine list API, sidebar rendering, session creation.

### Anti-Patterns to Avoid

- **Running Docker with --privileged + --pid=host:** Destroys container isolation. Never do this for a user-facing app.
- **Mounting Docker socket into the container:** Grants full Docker API access. Security anti-pattern unless absolutely necessary.
- **Storing "This Machine" in the database:** Creates migration complexity, potential for deletion, and confusing state (what if the user deletes it?).
- **Using a separate sidecar process on the host:** Breaks the "single docker compose up" requirement. Adds operational burden.
- **Different WebSocket endpoints for local vs remote terminals:** Unnecessary. The same `/ws/terminal/{session_id}` endpoint works for both; the backend just uses a different transport.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SSH to Docker host | Custom networking solution | `host.docker.internal` + AsyncSSH | Docker's built-in host resolution. Works on Windows (Docker Desktop), macOS, and Linux (with extra_hosts). |
| PTY management (native) | Raw fork/exec from scratch | Python `pty.openpty()` + `asyncio` event loop readers | stdlib handles PTY pair creation; asyncio handles non-blocking I/O. |
| Tmux session management | Custom process tracker | Existing tmux CLI commands via subprocess | Same commands as SSH path, just different transport. |
| Terminal resize | Custom signal handling | `TIOCSWINSZ` ioctl (native) or `channel.set_terminal_size` (SSH) | Standard POSIX terminal resize mechanism. |
| Docker detection | Complex heuristic | `os.path.exists("/.dockerenv")` | Simple, reliable. Docker always creates this file. |

## Common Pitfalls

### Pitfall 1: Docker host.docker.internal Not Available on Linux Docker Engine

**What goes wrong:** `host.docker.internal` resolves automatically on Docker Desktop (Windows/macOS) but NOT on plain Docker Engine on Linux.
**Why it happens:** Docker Desktop injects the DNS entry. Docker Engine does not.
**How to avoid:** Always add `extra_hosts: ["host.docker.internal:host-gateway"]` in docker-compose.yml for both the `app` and `app-dev` services. This works on ALL platforms.
**Warning signs:** Container can't resolve `host.docker.internal` at runtime.

### Pitfall 2: SSH Key Permissions in Docker Volume Mount

**What goes wrong:** SSH keys mounted from Windows host via Docker volume have wrong permissions (too open). SSH refuses to use them.
**Why it happens:** Windows filesystem doesn't support Unix permissions. Volume mounts from Windows always show 777.
**How to avoid:** Copy SSH keys into the container and fix permissions at container startup, OR use `ssh-keygen` to generate a container-specific keypair and add the public key to the host's `authorized_keys`.
**Warning signs:** AsyncSSH or OpenSSH errors about "permissions too open" on key files.

### Pitfall 3: pty Module Not Available on Windows

**What goes wrong:** Python's `pty` module is Unix-only. If someone tries to run the backend natively on Windows, PTY creation fails.
**Why it happens:** Windows doesn't have POSIX pseudo-terminals.
**How to avoid:** The native PTY path is only for Linux/macOS hosts. On Windows, the backend always runs in Docker (Linux container), so it uses the SSH-to-host path. Document this clearly. Gate the native PTY code behind a platform check.
**Warning signs:** `ImportError` or `AttributeError` when importing `pty` or `fcntl` on Windows.

### Pitfall 4: Race Condition Between "This Machine" and DB Machines

**What goes wrong:** If frontend fetches machines before backend startup completes, the local machine might not appear, or it might appear before remote machines are connected.
**Why it happens:** Local machine is injected at API level, not from DB. Timing differs.
**How to avoid:** Local machine injection is synchronous (it's always there, always "online"). Frontend treats it as always-first in the list. No race possible if done correctly.
**Warning signs:** "This Machine" flickering in sidebar on first load.

### Pitfall 5: Terminal Session TerminalSession.machine_id FK Constraint

**What goes wrong:** Creating a TerminalSession with `machine_id="local"` fails because there's no Machine row with that ID in the database. Foreign key constraint violation.
**Why it happens:** TerminalSession has `ForeignKey("machines.id")` and "This Machine" isn't in the machines table.
**How to avoid:** Two options: (A) Make machine_id nullable or remove FK constraint, storing "local" as a string. (B) Insert a synthetic Machine row with a well-known UUID on startup. Option A is cleaner -- make the FK optional and handle "local" as a special case.
**Warning signs:** IntegrityError on session creation for local machine.

### Pitfall 6: Host SSH Server Not Running

**What goes wrong:** In Docker mode, the container tries to SSH to the host but there's no SSH server running on the host.
**Why it happens:** Not all hosts have sshd enabled (especially Windows without OpenSSH Server, or macOS without Remote Login).
**How to avoid:** Clear error message and setup instructions. On first launch, detect SSH availability and guide user through enabling it. Consider making this a setup wizard step. The system should still work for remote machines even if local SSH fails.
**Warning signs:** "Connection refused" when connecting to host.docker.internal:22.

### Pitfall 7: WSL2 Networking Complexity on Windows

**What goes wrong:** `host.docker.internal` on Docker Desktop for Windows (WSL2 backend) resolves to the Windows host IP, not the WSL2 VM. SSH server in WSL2 isn't reachable via this address.
**Why it happens:** Docker Desktop uses a Hyper-V virtual network. host.docker.internal points to the Windows host.
**How to avoid:** The SSH server should run on the Windows host (Windows OpenSSH), not in WSL2. Alternatively, configure the SSH server in WSL2 to be accessible and set LOCUS_HOST_SSH_HOST to the WSL2 IP. Document both paths.
**Warning signs:** SSH connects to Windows (PowerShell) instead of expected bash/tmux environment.

### Repo Discovery for Local Machine

```python
async def scan_local_repos(scan_paths: list[str]) -> list[str]:
    """Scan local filesystem for git repositories."""
    repos = []
    for scan_path in scan_paths:
        result = await local_machine_manager.run_command(
            f"find {scan_path} -maxdepth 2 -name .git -type d 2>/dev/null"
        )
        for line in result.strip().split("\n"):
            if line:
                repo_path = line.rstrip("/")
                if repo_path.endswith("/.git"):
                    repo_path = repo_path[:-5]
                repos.append(repo_path)
    return repos
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| SSH-to-localhost only | Transport abstraction (SSH in Docker, PTY natively) | This phase | Local machine works in both deployment modes |
| All machines in DB | Synthetic "This Machine" + DB machines | This phase | Local machine always present, can't be deleted |
| SSH-only terminal sessions | Dual transport terminal sessions | This phase | Same UX for local and remote |

## Open Questions

1. **Host SSH server setup UX**
   - What we know: Docker mode requires an SSH server running on the host. Windows needs OpenSSH Server enabled. macOS needs "Remote Login" enabled. Linux usually has sshd.
   - What's unclear: How much setup guidance to provide in the UI. Should there be a setup wizard step?
   - Recommendation: Add a "Local Machine Setup" section in Settings that checks connectivity and provides platform-specific instructions. Don't block the app from launching -- just show a warning if local SSH is not available.

2. **repo_scan_paths for local machine**
   - What we know: Remote machines store repo_scan_paths in the Machine DB row. Local machine has no DB row.
   - What's unclear: Where to store local repo_scan_paths.
   - Recommendation: Add a `LOCUS_LOCAL_REPO_SCAN_PATHS` env var (comma-separated), and/or store in the existing settings/config table. The Settings page already exists from Phase 1.

3. **TerminalSession FK constraint**
   - What we know: TerminalSession.machine_id is a FK to machines.id. Local machine has no machines row.
   - What's unclear: Best migration strategy.
   - Recommendation: Make machine_id a nullable string field (dropping the FK), or use a sentinel UUID for "local". Dropping the FK is cleaner since it allows flexible machine references without migration complexity on every new machine type.

4. **Windows host: tmux availability**
   - What we know: tmux is not native to Windows. If Docker SSH connects to Windows OpenSSH, there's no tmux.
   - What's unclear: Whether to support bare PTY (no tmux) on Windows hosts, or require WSL2.
   - Recommendation: If the SSH connection lands on Windows (detected by running `uname` or checking default shell), fall back to a bare PowerShell/CMD session without tmux wrapping. Document that tmux features (session persistence, detach/reattach) require WSL2 or a Unix host. The core use case is Linux/macOS hosts and WSL2 on Windows.

5. **SSH key generation and management for Docker-to-host**
   - What we know: The container needs an SSH key that's authorized on the host.
   - What's unclear: Whether to auto-generate a key pair at first launch or require manual setup.
