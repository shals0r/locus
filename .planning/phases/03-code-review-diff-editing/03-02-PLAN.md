Create backend/app/services/review_service.py with:

1. `ReviewProvider` ABC with these abstract methods:
   - `async def get_mr_metadata(self, mr_id: str) -> dict` - Fetch MR/PR title, author, status, reviewers, pipeline, diff_refs
   - `async def get_mr_diff(self, mr_id: str) -> dict` - Fetch changed files with per-file diffs (unified diff text per file)
   - `async def get_mr_comments(self, mr_id: str) -> list[dict]` - Fetch existing comments as threads
   - `async def post_review(self, mr_id: str, comments: list[dict], event: str, body: str) -> dict` - Post review with comments + event
   - `async def reply_to_comment(self, mr_id: str, thread_id: str, body: str) -> dict` - Reply to existing thread
   - `async def approve(self, mr_id: str) -> dict` - Approve the MR/PR
   - `async def request_changes(self, mr_id: str, body: str) -> dict` - Request changes (GitLab: post comment + unapprove)

2. `get_review_provider(source_type: str, credential_data: dict, project_info: dict) -> ReviewProvider` factory function. `source_type` is "github" or "gitlab". `credential_data` is the decrypted token. `project_info` contains owner/repo or project_id.

Create backend/app/services/github_review.py with `GitHubReviewProvider(ReviewProvider)`:
- Constructor takes: `token: str, owner: str, repo: str`
- Uses httpx.AsyncClient for all API calls to https://api.github.com
- `get_mr_diff`: GET /repos/{owner}/{repo}/pulls/{pr_number}/files -- returns list of files with patches
- `get_mr_metadata`: GET /repos/{owner}/{repo}/pulls/{pr_number} -- extract title, user.login, state, requested_reviewers, head/base refs
- `get_mr_comments`: GET /repos/{owner}/{repo}/pulls/{pr_number}/comments -- group by in_reply_to_id to form threads
- `post_review`: POST /repos/{owner}/{repo}/pulls/{pr_number}/reviews -- atomic: includes `event` (APPROVE/REQUEST_CHANGES/COMMENT), `body`, and `comments` array with `path`, `line`, `side` (use newer line+side API, NOT position)
- `reply_to_comment`: POST /repos/{owner}/{repo}/pulls/{pr_number}/comments/{comment_id}/replies
- `approve`: Shorthand for post_review with event=APPROVE
- `request_changes`: Shorthand for post_review with event=REQUEST_CHANGES

Create backend/app/services/gitlab_review.py with `GitLabReviewProvider(ReviewProvider)`:
- Constructor takes: `token: str, project_id: str, gitlab_url: str = "https://gitlab.com"`
- Uses httpx.AsyncClient for all API calls to {gitlab_url}/api/v4
- `get_mr_diff`: GET /projects/{id}/merge_requests/{iid}/changes -- extract diffs per file
- `get_mr_metadata`: GET /projects/{id}/merge_requests/{iid} -- extract title, author.username, state, reviewers, pipeline.status, diff_refs (critical for comment positioning)
- `get_mr_comments`: GET /projects/{id}/merge_requests/{iid}/discussions -- already threaded (notes within discussions)
- `post_review`: GitLab has NO atomic review concept. Loop through comments, creating individual POST /projects/{id}/merge_requests/{iid}/discussions for each. Each discussion needs a `position` object with `base_sha`, `head_sha`, `start_sha`, `position_type: "text"`, `new_path`, `new_line`. Then if event is APPROVE, call approve separately.
- `reply_to_comment`: POST /projects/{id}/merge_requests/{iid}/discussions/{discussion_id}/notes
- `approve`: POST /projects/{id}/merge_requests/{iid}/approve
- `request_changes`: Post a comment, then POST /projects/{id}/merge_requests/{iid}/unapprove (GitLab has no formal "request changes")

For credential lookup: providers receive pre-decrypted token. The API layer (Plan 05) handles credential lookup and decryption using the existing crypto.decrypt_value pattern from services/crypto.py.

   - System prompt: "You are a senior code reviewer. Analyze the following git diff and provide specific, actionable review comments.    - Parse response: extract content[0].text, json.loads it
   - Add a UUID id field to each annotation if not present
   - Handle errors: if llm_api_key is empty, raise ValueError("LLM API key not configured"). If response parsing fails, return empty list with a logged warning.

   - Append context as a system message prefix.
   - Return the assistant's text response.
   - Use same httpx + Anthropic API pattern as review_diff.
