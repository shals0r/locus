  - `async def execute(self, source: IntegrationSource)`: Concrete.   - **`_elevate_tier_if_mentioned(self, item: dict, source: IntegrationSource) -> dict`**: Calls _is_mentioned(). If True and current tier_hint is "prep" or "follow_up" or None, elevate to "respond". If current tier_hint is "review", elevate to "respond". If already "now" or "respond", leave as-is. Returns the (possibly modified) item dict. This implements FEED-05: items where user is @mentioned get elevated urgency.

Implement all four adapters following the RESEARCH.md examples exactly. Each extends BasePollingAdapter and implements `poll()`.

**github_adapter.py** (GitHubAdapter):
- Config expects: token, repos (list of "owner/repo"), github_username (for mention detection)
- Polls: open PRs (GET /repos/{owner}/{repo}/pulls?state=open)
- For each PR: external_id="pr:{repo}:{number}", title="PR #{num}: {title}", snippet=body[:100], url=html_url, source_icon="github"
- Tier classification: requested_reviewers -> "review", draft -> "follow_up", else "prep"
- Mention detection: include PR body text and review comments for @mention scanning by base class
- Uses httpx.AsyncClient with Bearer auth and timeout=30

**gitlab_adapter.py** (GitLabAdapter):
- Config expects: token, base_url (default "https://gitlab.com"), project_ids (optional), gitlab_username (for mention detection)
- Polls: MRs assigned_to_me and review_requested (GET /api/v4/merge_requests?scope=...)
- external_id="mr:{project_id}:{iid}", title="MR !{iid}: {title}", source_icon="gitlab"
- Tier: review_requested -> "review", assigned -> "prep"
- Mention detection: include MR description for @mention scanning
- Uses PRIVATE-TOKEN header auth

**jira_adapter.py** (JiraAdapter):
- Config expects: base_url, email, api_token, jira_username (for mention detection)
- Polls: JQL "assignee = currentUser() OR watcher = currentUser() ORDER BY updated DESC"
- external_id="issue:{key}", title="{key}: {summary}", source_icon="jira"
- Tier: Highest->"now", High->"respond", Medium->"prep", Low/Lowest->"follow_up"
- Mention detection: include issue description and latest comment body for @mention scanning
- Uses HTTP basic auth (email:api_token)

**calendar_adapter.py** (GoogleCalendarAdapter):
- Config expects: access_token, refresh_token (manual setup), calendar_id (default "primary")
- Polls: events from now to 24h ahead (GET /calendar/v3/calendars/{id}/events)
- external_id="event:{event_id}", title=event summary, snippet="Starts: {time}", source_icon="calendar"
- Tier: always "prep" (calendar events are planning items)
- Token refresh: if 401, use refresh_token to get new access_token via Google OAuth token endpoint, update source config
- Per research: accept pre-generated tokens; full OAuth flow is Phase 4

Register all four in scheduler.py's get_adapter() registry.

All adapters: wrap httpx calls in try/except, log errors, return empty list on failure (never crash the scheduler job). Each adapter's poll() returns items with enough body/description text that _is_mentioned() in the base class can detect @mentions.
